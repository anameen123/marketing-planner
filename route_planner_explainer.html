🗺️ Route Planner — Random clinic scatter + corridor levels

Member sets Start and End + how many stops they want in between (here: 5). The straight dashed line gets split into 5 equal slices. Each slice becomes a rectangle extending width mi each side. Clinics/MVAs anywhere on the map are bucketed into the level whose rectangle they fall inside. Anything outside the corridor is ignored unless the user expands the width slider.

What the picture shows: 5 stops requested between Start and End. The dashed line is split into 5 equal slices of length d. Each slice becomes a corridor extended width mi on each side. 📐 The geometry alone — no pins, just dimensions

Shows exactly how the straight line gets sliced into N rectangles and how each rectangle's dimensions relate to d (segment length) and w (corridor width).

Per slice: length along the route = d. Width perpendicular = w, split evenly above and below the straight line (so w/2 on each side). The line passes through the exact horizontal center of each rectangle.

🚧 The multi-pick deroute problem (the thing you flagged)

If a level has clinics on both sides of the line, picking both forces a perpendicular zigzag. The distance from one to the other is mostly wasted — it doesn't progress toward End. Compare:

✅ Both on same side — small detour
⛔ Opposite sides — big wasted zigzag

💡 The fix — a deroute budget per level

Calculate the extra distance each multi-pick combo adds beyond the straight line. If it exceeds a configurable budget (e.g. 0.5 × d), block the second pick or warn loud.

So the member experience is: they can multi-pick freely while everything stays cheap. As soon as their picks would force a big zigzag (opposite-side pair, or two pins very far from the line), the planner warns them: "Picking both adds X mi of detour — confirm or pick just one?" User confirms or backs off.

🎨 Pin colors

Green · > 3 weeks since last visit (or never) — TOP PRIORITY
Orange · 2–3 weeks since last visit — secondary
Red · < 2 weeks ago — EXCLUDED from recs
Purple S/E · Start & End pins

📐 The math

L = straight-line(S → E)
d = L / N (here N = 5)
width = user-tunable (1 mi/side default)
in_level_i(pin):
  projection onto line ∈ [i·d, (i+1)·d]
  AND ⊥ distance < width
deroute(pin) = ⊥ distance from line
